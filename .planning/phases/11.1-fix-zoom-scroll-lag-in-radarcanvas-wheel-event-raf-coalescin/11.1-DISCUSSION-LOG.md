## Wheel-Event Coalescing (D-01..D-04)

| Option | Description | Selected |
|--------|-------------|----------|
| rAF-coalesced setViewport | Accumulate wheel events into a pending ref, flush one setViewport per rAF | ✓ |
| Time-based throttle (every 16ms) | setTimeout-driven; decoupled from render loop | |
| Leading-edge debounce | Fire immediately then ignore for 16ms window | |
| Let events flow, move viewport to ref (no state) | Eliminates React re-renders entirely | |

**Auto-selected:** rAF-coalesced setViewport with accumulated `deltaY`.
**Reasoning:** rAF-coalescing is the canonical browser pattern for "don't do work faster than the display can show it." Time-based throttles drift against rAF and can cause frame-skip. Leading-edge debounce feels unresponsive. Moving viewport to a ref would require rewiring `RadarMinimap` and any test that reads the state — more scope than needed.

## Viewport Store Writeback (D-05..D-07)

| Option | Description | Selected |
|--------|-------------|----------|
| Keep store slot + rAF-throttle writeback | Defensive belt-and-suspenders with D-01 | ✓ |
| Remove viewport from store; make RadarMinimap use context | Larger API surface change | |
| Keep writeback unchanged; rely on D-01 alone | Minimal change, still correct | |

**Auto-selected:** Keep store slot + rAF-throttle writeback.
**Reasoning:** D-01 alone would likely fix the issue (since `setViewport` only fires once per rAF, `storeSetViewport` also only fires once per rAF). But the `useEffect([viewport])` pattern depends on React's scheduler — if React batches multiple `setViewport` calls into one re-render, the writeback still fires exactly once, which is fine. The belt-and-suspenders throttle guards against any future caller that writes viewport via a non-coalesced path.

## Folder-Hull Caching (D-08..D-11)

| Option | Description | Selected |
|--------|-------------|----------|
| Cache keyed on settledAt (+ optional positionsGenerationCounter) | Invalidate only on real position changes | ✓ |
| Cache keyed on nodeId-list hash | More work per frame to compute the hash | |
| Per-directory cache with per-node invalidation | Complex; invalidation granularity not worth it | |
| No cache; accept the cost | Phase boundary says we're fixing this | |

**Auto-selected:** Coarse single-cache invalidation on settledAt bump.
**Reasoning:** 5k nodes → ~200 directories → ~200 polygonHull calls per frame at O(n log n) each. Caching eliminates this entirely when the graph is settled (which is the common case — zoom happens on settled graphs). Coarse invalidation (whole cache evicts on any position change) is correct-by-construction: if any node moved, all hulls could potentially change, but in practice `commitSettledPositions` fires only at real settle events, not per-tick.

| Option | Description | Selected |

| Full PerformanceObserver harness | Overkill for this scope | |
| On-screen HUD overlay component | More UI code; out-of-scope |  |
| No diagnostic | Leaves no way to measure the fix in prod build | |

**Auto-selected:** localStorage-gated rolling log.
**Reasoning:** Tauri v2's WebKitGTK DevTools lacks the Performance panel, so the only way the user can empirically measure wheel-burst frame cost is an in-app diagnostic. Gating on `localStorage.radarPerfDebug` keeps it zero-cost in production.
