# Phase 11.1: Fix zoom-scroll lag in RadarCanvas - Context

<domain>
## Phase Boundary

Eliminate the user-reported zoom-scroll lag in the Radar view. On a settled graph with ~5k nodes, wheel-driven zoom in/out causes perceptible UI stutter (confirmed during Phase 11 manual smoke). The lag is **not** a Phase 11 regression — when `isSimulatingRef.current === false` the hot-path gate short-circuits and the render loop is byte-identical to Phase 7. The root cause is pre-existing perf debt in three independent loci:

1. **Wheel-event amplification** — `useCanvasZoomPan.onWheel` calls `setViewport(...)` directly. Modern trackpads fire wheel events at 120–240 Hz. Each call triggers a full React re-render of RadarCanvas, recomputes `useMemo`s, re-runs the `stateRef` mirror effect, and marks the canvas dirty. rAF caps at 60 Hz so the event rate outpaces the render loop and the React work piles up.
2. **Unnecessary Zustand writeback on every wheel event** — `storeSetViewport(viewport)` in a `useEffect([viewport])` pushes the viewport into `radarStore` on every setViewport call. `RadarMinimap.tsx:44` subscribes to `radarStore.viewport`, so every wheel event also re-renders the minimap.
3. **Per-frame folder-hull recomputation** — `drawFolderHulls` calls `polygonHull(paddedPoints)` (d3-polygon, O(n log n)) for *every* directory *every* frame, even when node positions haven't moved since `settledAt`. At zoom levels where many directory hulls are visible, this is a non-trivial fraction of the frame budget on a 5k-node graph.

This phase is performance-only. No visual change. No new capability. No new requirement IDs. The user-visible success criterion is: **"wheel-zoom on a 5k-node settled graph in the Tauri prod build feels responsive — no perceptible lag during a 3-second continuous scroll burst."**

Out of scope: moving rendering to OffscreenCanvas (Phase 14), multi-layer compositing (Phase 14), re-enabling drag-to-pin (carried forward as a deferred user-accepted deviation from Phase 11), SharedArrayBuffer for worker transport (Phase 11 deferred).

</domain>

<decisions>
## Implementation Decisions

### Wheel-Event Coalescing
- **D-01:** Coalesce wheel events through `requestAnimationFrame` in `src/hooks/useCanvasZoomPan.ts`. On each `wheel` event: capture `deltaY`/`clientX`/`clientY` into a mutable ref (`pendingWheelRef`); if no rAF is scheduled, schedule one; inside the rAF callback, read the accumulated pending state, compute the new viewport once, clear the pending state, and call `setViewport(...)` exactly once per animation frame. Net effect: regardless of wheel-event rate (60Hz trackball, 120Hz trackpad, 240Hz gaming mouse), at most one `setViewport` per rAF.
- **D-02:** Accumulate `deltaY` from multiple events within a rAF window by summing — NOT by taking the last event only. Multi-event accumulation preserves the "faster scroll = faster zoom" feel and means nothing is dropped on the floor. Zoom factor is applied once at rAF time: `factor = deltaY < 0 ? ZOOM_IN_FACTOR ^ n : ZOOM_OUT_FACTOR ^ n` where `n` is the number of coalesced events (or equivalently a single `deltaY`-proportional factor — planner picks the exact math).
- **D-03:** Cursor-anchored zoom is preserved: the accumulated pending state captures the most-recent `cursorX`/`cursorY` (last event wins for the anchor point). Re-computing the pan offset inside the rAF callback against the final zoom factor gives a single smooth motion toward the cursor's most-recent position.
- **D-04:** On unmount, the scheduled rAF must be canceled (`cancelAnimationFrame`) to avoid a stray `setViewport` call on a dead component.

### Viewport Store Writeback
- **D-05:** Keep `radarStore.viewport` as a store slot — it's a legitimate subscription surface used by `RadarMinimap.tsx`. Do NOT break the store contract by removing or relocating `viewport`.
- **D-06:** Throttle the `storeSetViewport(viewport)` writeback through rAF (same cadence as D-01). Mechanism: wrap the writeback in its own pending-ref + rAF coalescer so the minimap (and any future subscriber) re-renders at most once per frame during a wheel burst. Since D-01 already caps `setViewport` at 1 call per rAF, this may be redundant — but it's cheap to add defensively and guards against any future caller that bypasses `setViewport` coalescing.
- **D-07:** Do NOT use `useDeferredValue` or React 19 `useTransition` for the writeback. These add scheduler complexity for no measurable gain when rAF coalescing already solves the problem. Keep the writeback mechanism imperative and ref-based.

### Folder-Hull Caching
- **D-08:** Cache folder hulls in `src/views/Radar/GraphRenderer.ts` (or a new `src/views/Radar/hullCache.ts` module — planner's choice) keyed on `(settledAt | positionsGenerationCounter)`. When the cache key is unchanged, return the cached `{ hull, centroid, padded, smoothPath }` bundle for each directory. When the key changes (new topology settle, pin move, force-config restart), invalidate the entire cache.
- **D-09:** Cache invalidation contract is coarse — the WHOLE cache evicts on any position change, not per-directory. 5k nodes produce maybe ~200 directories, so rebuilding all hulls on invalidation is ~200 × `polygonHull` calls. That's fine when it happens only on real topology events, not every frame.
- **D-10:** The cache key is exposed via `useRadarStore.getState().settledAt` (the existing ms-epoch field) + a separate `positionsGenerationCounter` that bumps whenever `commitSettledPositions` or `pinNode` fires. Planner decides whether to add the counter to the store or derive it from `settledAt` alone (settledAt alone may suffice — pin moves already set settledAt via the hook's commit path — so the simpler single-key variant is preferred unless research shows pin moves bypass the settle path).
- **D-11:** Zoom changes do NOT invalidate the hull cache. Hulls are computed in world-space; the viewport transform is applied by the Canvas 2D context after the hull path is constructed. Zooming just re-uses the cached smooth-spline path at a different matrix transform.

### Profiling & Telemetry
- **D-12:** Add a dev-only diagnostic: wrap the rAF render callback in `performance.now()` brackets and emit a rolling p95/max to a `console.log` (or optional on-screen overlay) when a `localStorage` debug flag is set (`localStorage.radarPerfDebug = '1'`). Zero production overhead. Gives the user and future debuggers a way to measure the fix's impact empirically inside the Tauri build (which has no DevTools Performance panel).
- **D-13:** Do NOT add a full `PerformanceObserver` harness or a diagnostic panel component. The rolling log is sufficient for this phase's scope.

### Testing Strategy
- **D-14:** Add a Vitest unit test for the wheel coalescer: stub `requestAnimationFrame` / `cancelAnimationFrame` (jsdom supports this pattern — drive synchronously via a flush helper); dispatch N `wheel` events; assert `setViewport` is called exactly once after flushing the rAF queue; assert the accumulated `deltaY` was honored.
- **D-15:** Add a Vitest unit test for the hull cache: call `drawFolderHulls` or the extracted cache accessor twice with the same `settledAt`; assert `polygonHull` is called only once per directory (spy on `d3-polygon.polygonHull`). Change `settledAt`; call again; assert the cache rebuilt.
- **D-16:** Manual smoke: user confirms the zoom feels responsive on a 5k-node graph in the Tauri prod build. This is the primary acceptance criterion — automated tests are regression guards, not user-experience validators.

### Carry-Over from Phase 11
- **D-17:** RadarCanvas hot-path Float32Array read (Phase 11 D-25/D-26) is unchanged. The zoom-scroll fix does NOT touch the simulating branch; the gate at `if (simulating && live.positions.byteLength > 0 && ...)` continues to short-circuit for settled graphs.
- **D-18:** Worker architecture (Phase 11 D-01..D-24) is untouched. The zoom lag is a main-thread issue; the worker is irrelevant here.
- **D-19:** Drag-to-pin remains the user-accepted Phase 11 deviation — drag pans the canvas. Phase 11.1 does not re-enable pin gestures. The pin/unpin worker protocol (D-20 in Phase 11) still works; there's just no UI surface invoking it in this phase.

- rAF-dirty-flag rendering pattern (Phase 4, preserved through Phase 11) — `dirtyRef.current = true` + single long-lived rAF loop. Phase 11.1 reuses this shape; the wheel coalescer is a separate rAF but follows the same discipline.
- Hot-path refs over store subscriptions — `pendingWheelRef` fits this pattern; wheel events never touch Zustand until the rAF flush.
- One Zustand store per domain — `radarStore.viewport` stays a store slot (D-05) because `RadarMinimap` legitimately consumes it.
- Test co-location — `__tests__/` folders next to modules.
- Performance budget: Phase 7 D-23 says "5,000 nodes + edges at 60fps during pan/zoom". Phase 11.1 delivers this on user hardware.

- No backend changes. No new deps. No Tauri config changes. No protocol changes.
