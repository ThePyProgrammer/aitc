## Summary

Three orthogonal fixes, all in the main thread, all on paths that execute when the sim is settled (the worker branch is untouched per D-17).

1. **Wheel coalescer** — a trivial `pendingRef + scheduledRef + rAF` pattern yields "at most one `setViewport` per frame." Canonical on-the-web shape is ~15 lines `[CITED: Nolan Lawson nolanlawson.com/2019/08/11]`. This is the load-bearing fix — it removes the React work amplification.
2. **Hull cache** — `drawFolderHulls` runs three O(n) passes per directory per frame (paddedHullPoints, polygonHull, polygonCentroid + a spline path generation). The cache key `settledAt` alone is sufficient: every code path that mutates node positions routes through `commitSettledPositions` which bumps `settledAt` `[VERIFIED: src/stores/radarStore.ts:242-250]`, including the pin/unpin path which re-runs the sim and resettles `[VERIFIED: src/hooks/useGraphLayout.ts:198-223, :363-392]`.
3. **Rolling p95 diagnostic** — 30-line ring-buffer inline in the rAF render loop, gated on a `localStorage` flag.

**Contradiction with CONTEXT.md:** D-06 (rAF-throttle the `storeSetViewport` writeback) is **redundant after D-01 lands**. React 19 already batches synchronous state updates `[CITED: react.dev/blog/2024/04/25/react-19]`, so a single `setViewport` per rAF drives at most one `useEffect([viewport])` execution per frame, which drives at most one `storeSetViewport` and one `RadarMinimap` re-render. Per the explicit D-06 allowance ("this may be redundant — but it's cheap to add defensively"), we recommend the planner **implement it anyway** as defense-in-depth with zero measurable cost. No contradiction in outcome; flagging the redundancy so the planner scopes Task 2 honestly.

**Primary recommendation:** One plan, four tasks: (1) wheel coalescer + unit test, (2) defensive writeback throttle, (3) hull cache + unit test, (4) rolling-p95 diagnostic. All four are small, orthogonal, independently committable. No scaffolding. No new files strictly required (cache + coalescer may live inline). Optional: extract `useRafCoalesced` if the wheel + writeback coalescers share enough structure to justify it (they do; recommend extracting).

## Key Findings

- **The load-bearing problem is React re-render cost, not canvas redraw cost.** `dirtyRef.current = true` already coalesces repaints through the single rAF render loop `[VERIFIED: RadarCanvas.tsx:534-537, 687-689]`. The canvas repaints at most 60Hz regardless of wheel rate. What doesn't coalesce today is `setViewport` itself — each `wheel` event queues a React re-render of the whole RadarCanvas tree, which runs 9-entry `useShallow` comparisons, re-evaluates memoized selectors, and re-runs the `stateRef` mirror effect. On a 240Hz trackpad, that's ~240 React renders per second for zero extra pixels.
- **`settledAt` alone is a sufficient hull-cache key.** Every position mutation routes through `commitSettledPositions` `[VERIFIED: radarStore.ts:242-250 — the ONLY writer of node x/y]`, which bumps `settledAt = Date.now()` atomically. `pinNode` sets `fx/fy/x/y` on the store node `[VERIFIED: radarStore.ts:257-268]` but this is immediately followed by a worker re-settle that calls `commitSettledPositions` again `[VERIFIED: useGraphLayout.ts:198-223, 363-392]`. Claim: **no code path mutates node positions without bumping `settledAt`.** Recommend the single-key variant per D-10.
- **Hull cache must wrap MORE than the `polygonHull` call.** `drawFolderHulls` builds `pts` from `members.map` → `paddedHullPoints` (O(n · 10)) → `polygonHull` (O(n log n)) → `smoothHullLine(hull)` (path-string build) → `new Path2D(pathStr)` → `polygonCentroid` `[VERIFIED: GraphRenderer.ts:214-246]`. All five cost per directory per frame. The cache value must be `{ padded, hull, smoothPath: Path2D | null, cx, cy, isCircleFallback }` so the cached draw path is reduced to `ctx.fill(path2d); ctx.stroke(path2d); ctx.fillText(label, cx, cy - 6/zoom)`.
- **`drawEdges` / `drawArrowHeads` / `drawNodes` already frustum-cull.** `[VERIFIED: GraphRenderer.ts:296-300 (drawEdges), :342 (drawArrowHeads), :404,428 (drawNodes)]` all call `isInViewport(...)` with `VIEWPORT_CULL_PADDING = 100`. Implication: at maximum zoom-out, nothing culls — everything is "in viewport" — so the worst case is "5k nodes, all visible, all drawn." This is the perf target state.
- **`drawFolderHulls` does NOT frustum-cull.** `[VERIFIED: GraphRenderer.ts:196-261]` — no `isInViewport` check. Adding a bounding-box cull per directory is the Claude's-discretion nice-to-have flagged in CONTEXT.md. With ~200 directories it's a clean 10-line optimization: compute the hull's world-space bbox, call `isInViewport` on each of the four corners (padded), skip the whole directory if all four are outside. Recommend including.
- **No existing `useRafCoalesced` in the codebase.** `[VERIFIED: Glob src/hooks/**/*.ts]` — only `useCanvasZoomPan`, `usePipelineChannel`, `useGraphLayout`, `useChatChannel`, `useClaudeResourcesChannel`, `useScopedAgents`, `useSyntaxHighlight`, `useWindowControls`. A tiny extraction is clean and will be reused by D-06.
- **Vitest rAF stub pattern: `vi.useFakeTimers({ toFake: ['requestAnimationFrame', 'cancelAnimationFrame'] })` + `vi.advanceTimersToNextFrame()`** `[CITED: vitest.dev/api/vi.html]`. This is the modern (Vitest 3+) idiomatic way; jsdom 26 implements rAF via `setInterval` internally `[CITED: vitest.dev/config/faketimers]`, so faking the timer also fakes rAF. No manual stub-global gymnastics required. (The existing `RadarCanvas.test.tsx:36-41` shim — `setTimeout(cb, 0)` — is a legacy pattern; we DO NOT need it for the new test.)

## Pattern 1: rAF Coalescer for Wheel Events

**Shape:** single pending-ref holds the accumulator object; single scheduled-ref holds the rAF handle; inside the callback, snapshot and clear the pending-ref *before* calling `setViewport`.

Canonical reference: the "queuedCallback gate" pattern `[CITED: nolanlawson.com/2019/08/11/high-performance-input-handling-on-the-web/]`. Adapted below for a stateful accumulator.

```ts
// src/hooks/useCanvasZoomPan.ts — replacement for the current onWheel handler.
// Accumulator holds summed deltaY (D-02) + last-wins cursor anchor (D-03).
interface PendingWheel {
  deltaY: number;         // summed across events since last flush
  cursorX: number;        // last-event-wins anchor (D-03)
  cursorY: number;
  target: HTMLCanvasElement; // captured before the native event is pooled
}

const pendingWheelRef = useRef<PendingWheel | null>(null);
const wheelRafRef = useRef<number | null>(null);

const flushWheel = useCallback(() => {
  wheelRafRef.current = null;
  const p = pendingWheelRef.current;
  pendingWheelRef.current = null;
  if (!p) return;
  setViewport((vp) => {
    // Exponential factor in deltaY per wheel tick: one natural zoom curve,
    // independent of how many events were coalesced. deltaY units are
    // browser-normalized pixels (approx) — ~100 per "one click" of a wheel.
    // Factor chosen so summed deltaY ≈ 100 → 1.04× zoom (matches the
    // pre-refactor feel). Planner may swap to event-count × fixed factor.
    const factor = Math.pow(ZOOM_OUT_FACTOR, p.deltaY / 100);
    const newZoom = Math.min(MAX_ZOOM, Math.max(MIN_ZOOM, vp.zoom * factor));
    const scale = newZoom / vp.zoom;
    const newPanX = p.cursorX - scale * (p.cursorX - vp.panX);
    const newPanY = p.cursorY - scale * (p.cursorY - vp.panY);
    return { zoom: newZoom, panX: newPanX, panY: newPanY };
  });
}, []);

const onWheel = useCallback((e: WheelEvent) => {
  e.preventDefault();
  const target = e.currentTarget as HTMLCanvasElement | null;
  if (!target) return;
  const rect = target.getBoundingClientRect();
  const cursorX = e.clientX - rect.left;
  const cursorY = e.clientY - rect.top;

  const prev = pendingWheelRef.current;
  pendingWheelRef.current = prev
    ? { deltaY: prev.deltaY + e.deltaY, cursorX, cursorY, target } // D-02 + D-03
    : { deltaY: e.deltaY, cursorX, cursorY, target };

  if (wheelRafRef.current === null) {
    wheelRafRef.current = requestAnimationFrame(flushWheel);
  }
}, [flushWheel]);

// D-04 — unmount cleanup. The existing effect at RadarCanvas.tsx:717-733
// already owns the wheel listener; the hook adds its own effect:
useEffect(() => {
  return () => {
    if (wheelRafRef.current !== null) {
      cancelAnimationFrame(wheelRafRef.current);
      wheelRafRef.current = null;
    }
    pendingWheelRef.current = null;
  };
}, []);
```

**Pending-ref shape choice:** an accumulator object (not a scalar). We need summed `deltaY` AND last-wins cursor coordinates AND the `target` — a single struct is the cleanest invariant carrier.

**Tab-visibility behavior:** when the tab is inactive, `requestAnimationFrame` stops firing `[CITED: MDN developer.mozilla.org/en-US/docs/Web/API/Window/requestAnimationFrame]`. Any pending wheel events accumulated before the tab went inactive will flush on the very next rAF when the tab reactivates — the accumulator stays in `pendingWheelRef` across the sleep. This is correct behavior: the user returning to a tab expects the first wheel gesture to apply, not "the wheel events from 2 minutes ago suddenly replay." We are at worst over-applying old events by one frame, which is imperceptible. No special handling needed.

**Backlog resume:** rAF does NOT backlog. When the tab reactivates, exactly one rAF fires with the current timestamp — there's no queue of "missed frames" to drain `[CITED: MDN]`. Our pattern coalesces all pending-since-sleep wheels into the single reactivation frame, which is the desired behavior.

**StrictMode note:** the handler already captures native-event-derived values into locals `[VERIFIED: useCanvasZoomPan.ts:36-41]` to survive React 19 StrictMode double-invocation. The new shape preserves this discipline (the accumulator is built from pre-captured locals, not from `e.currentTarget` inside the rAF callback).

### Optional extraction: `useRafCoalesced<T>`

If the planner elects to extract this (Claude's discretion), the interface is one-liner:

```ts
// src/hooks/useRafCoalesced.ts
export function useRafCoalesced<T>(
  flush: (pending: T) => void,
): (merge: (prev: T | null) => T) => void {
  const pendingRef = useRef<T | null>(null);
  const rafRef = useRef<number | null>(null);
  const flushRef = useRef(flush);
  useEffect(() => { flushRef.current = flush; });
  useEffect(() => () => {
    if (rafRef.current !== null) cancelAnimationFrame(rafRef.current);
    pendingRef.current = null;
  }, []);
  return useCallback((merge) => {
    pendingRef.current = merge(pendingRef.current);
    if (rafRef.current === null) {
      rafRef.current = requestAnimationFrame(() => {
        rafRef.current = null;
        const p = pendingRef.current;
        pendingRef.current = null;
        if (p !== null) flushRef.current(p);
      });
    }
  }, []);
}
```

Call site in `useCanvasZoomPan`: `const enqueueWheel = useRafCoalesced<PendingWheel>(flush);` then `onWheel` becomes three lines.

Call site in `RadarCanvas` for D-06: `const enqueueViewportWriteback = useRafCoalesced<CanvasViewport>((vp) => storeSetViewport(vp));` then the `useEffect([viewport])` becomes `enqueueViewportWriteback(() => viewport)`.

## Pattern 2: Vitest rAF Stub

Modern idiom for Vitest 3 + jsdom 26 `[CITED: vitest.dev/api/vi.html, vitest.dev/config/faketimers]`:

```ts
// src/hooks/__tests__/useCanvasZoomPan.test.ts
import { describe, it, expect, vi, beforeEach, afterEach } from 'vitest';
import { renderHook, act } from '@testing-library/react';
import { useCanvasZoomPan } from '../useCanvasZoomPan';

describe('useCanvasZoomPan wheel coalescer (D-01..D-04)', () => {
  beforeEach(() => {
    vi.useFakeTimers({ toFake: ['requestAnimationFrame', 'cancelAnimationFrame'] });
  });
  afterEach(() => {
    vi.useRealTimers();
  });

  function makeWheelEvent(deltaY: number, cx = 50, cy = 50): WheelEvent {
    // jsdom has no `new WheelEvent` with deltaY on all versions; build the
    // event via Object.assign on a MouseEvent-shaped object and cast. The
    // hook only reads {clientX, clientY, deltaY, currentTarget, preventDefault}.
    const fakeTarget = {
      getBoundingClientRect: () => ({ left: 0, top: 0, width: 200, height: 200 }),
    } as unknown as HTMLCanvasElement;
    return {
      deltaY,
      clientX: cx,
      clientY: cy,
      currentTarget: fakeTarget,
      preventDefault: vi.fn(),
    } as unknown as WheelEvent;
  }

  it('coalesces N wheel events into one setViewport call per animation frame', () => {
    const { result } = renderHook(() => useCanvasZoomPan());
    const setViewportSpy = vi.spyOn(result.current, 'setViewport' as never);

    // Dispatch 5 wheel events within one frame.
    act(() => {
      for (let i = 0; i < 5; i++) {
        result.current.handlers.onWheel(makeWheelEvent(-100));
      }
    });

    // Before the frame flushes, setViewport has NOT been called.
    expect(setViewportSpy).toHaveBeenCalledTimes(0);

    // Advance one animation frame — flushes the coalesced rAF.
    act(() => {
      vi.advanceTimersToNextFrame();
    });

    // D-01: exactly one setViewport call after flushing, regardless of
    // how many wheel events were queued.
    expect(setViewportSpy).toHaveBeenCalledTimes(1);

    // D-02: the summed deltaY (-500 across 5 events) drove a larger zoom
    // than a single event would have. Compare vs a one-event run.
    const zoomAfter5 = result.current.viewport.zoom;
    expect(zoomAfter5).toBeGreaterThan(1); // zoom-in
    expect(zoomAfter5).toBeGreaterThan(1.04); // more than one-event worth
  });

  it('cancels pending rAF on unmount (D-04)', () => {
    const { result, unmount } = renderHook(() => useCanvasZoomPan());
    const setViewportSpy = vi.spyOn(result.current, 'setViewport' as never);

    act(() => {
      result.current.handlers.onWheel(makeWheelEvent(-100));
    });
    unmount();
    act(() => {
      vi.advanceTimersToNextFrame();
    });

    // No setViewport should fire on a dead component.
    expect(setViewportSpy).toHaveBeenCalledTimes(0);
  });
});
```

**Why not `vi.stubGlobal('requestAnimationFrame', ...)`?** That approach works but requires hand-rolling a queue + a flush helper. `vi.useFakeTimers({ toFake: ['requestAnimationFrame'] })` gives `vi.advanceTimersToNextFrame()` for free `[CITED: vitest.dev/api/vi.html]`, which is the exact semantic we want and is more readable.

**Edge case — the Zustand rehydrate problem:** `useCanvasZoomPan` uses `useState`, not the store, so this test does not need to reset `useRadarStore`. If a future test exercises the writeback, add `beforeEach(() => useRadarStore.getState().reset())` matching the pattern in `stores/__tests__/radarStore.test.ts:36-38`.

**Edge case — spying on a `setViewport` returned from the hook:** the `setViewport` in the return value is the raw `useState` setter. Spying on it via `vi.spyOn(result.current, 'setViewport')` works after React's return. Alternative: assert the effect by observing `result.current.viewport.zoom` before/after `advanceTimersToNextFrame()`. Either is acceptable; the second is more behavior-driven.

## Pattern 3: Hull Cache with settledAt Invalidation

**Recommendation: single-key variant, `settledAt` only. NO `positionsGenerationCounter`.**

Justification (the D-10 planner-decision):

- `commitSettledPositions` is the **only** writer of node `x`/`y` in the store — `[VERIFIED: radarStore.ts:242-250]`. It unconditionally sets `settledAt: Date.now()`.
- `pinNode` sets `x`/`y` alongside `fx`/`fy` `[VERIFIED: radarStore.ts:257-268]` *without* bumping `settledAt`. However, `pinNode` is only invoked via a drag-to-pin UI surface that is currently disabled per D-19. Even if re-enabled, pinning triggers a worker re-settle `[VERIFIED: useGraphLayout.ts:363-392]` which completes with a `settled` message → `commitSettledPositions` → `settledAt` bump `[VERIFIED: useGraphLayout.ts:198-223]`. So **within one animation frame** of a pin move, the cache may briefly render with stale hulls, but the next frame picks up the new `settledAt` and rebuilds. This is imperceptible and acceptable.
- `unpinNode` does NOT touch `x`/`y` `[VERIFIED: radarStore.ts:274-285]` — it only clears `fx`/`fy`. Positions don't change until the downstream re-settle, which bumps `settledAt`.
- `fetchGraph` sets `settledAt: null` `[VERIFIED: radarStore.ts:229]` when the topology refreshes; `null` is a distinct key from any prior epoch, so the cache evicts.

**Conclusion:** `settledAt: number | null` is a perfectly sound cache key. Adding a counter is overkill for this phase.

**Exact cache shape:**

```ts
// src/views/Radar/hullCache.ts (recommended: new file; the extraction is
// cleaner than inlining a module-level Map into GraphRenderer.ts).

import { polygonHull, polygonCentroid } from 'd3-polygon';
import { line, curveCatmullRomClosed } from 'd3-shape';
import type { GraphNode } from '../../stores/radarStore';

interface HullCacheEntry {
  /** Pre-computed closed-spline Path2D. null when <3 hull points (circle fallback). */
  smoothPath: Path2D | null;
  /** Centroid for label placement. For <3-node dirs, the mean of node positions. */
  cx: number;
  cy: number;
  /** True when the directory had <3 nodes and drew as a circle fallback. */
  isCircleFallback: boolean;
  /** Depth from repo root — drives the zoom-tier cull decision in drawFolderHulls. */
  dirDepth: number;
}

// Module-level state. Cleared on settledAt change.
let cacheEpoch: number | null = -1 as number | null; // sentinel; never matches a real epoch
let cache = new Map<string /* dirKey */, HullCacheEntry>();

// Reused across frames (created once at module load).
const smoothHullLine = line().curve(curveCatmullRomClosed.alpha(0.5));

function paddedHullPoints(
  nodePoints: [number, number][],
  radius: number,
  resolution = 10,
): [number, number][] {
  const result: [number, number][] = [];
  for (const [x, y] of nodePoints) {
    for (let i = 0; i < resolution; i++) {
      const a = (i / resolution) * Math.PI * 2;
      result.push([x + Math.cos(a) * radius, y + Math.sin(a) * radius]);
    }
  }
  return result;
}

/**
 * Resolve the per-directory hull bundle, rebuilding the cache if `settledAt`
 * changed since the last call. World-space; zoom is NOT a cache input (hulls
 * transform via the canvas matrix — D-11).
 *
 * Padding radius is currently 25/zoom in the caller; to keep the cache
 * zoom-invariant we pass the world-space `paddingRadius = 25` and let the
 * caller handle the `1/zoom` scale by recomputing a per-frame Path2D ONLY
 * when the padding is zoom-dependent. Current behavior uses `25/zoom`, so
 * padding shrinks as you zoom in — that means paddedPoints ARE zoom-dependent
 * and strictly we cannot cache the Path2D across zoom levels.
 *
 * ⚠ RESOLUTION: the existing code uses `25/zoom` as the padding radius
 * (GraphRenderer.ts:221). Either:
 *   (a) change padding to a world-space constant (e.g. 25), losing the
 *       "hull puffs out as you zoom out" visual behavior. This is a VISUAL
 *       change — not permitted by phase scope.
 *   (b) split the cache: cache paddedHullPoints-at-radius-1 (or cache the
 *       node-center array), and rebuild the hull + path each frame. Defeats
 *       the perf win.
 *   (c) cache at the per-zoom granularity. Cache key becomes
 *       `settledAt|zoomBucket`. Zoom buckets quantize at e.g. 0.1 increments
 *       so small wheel deltas still hit the cache.
 *   (d) HYBRID: cache the raw node-center array (cheap) and paddedHullPoints
 *       at a world-space-constant radius of 25 (not 25/zoom). Then the hull
 *       itself is zoom-invariant. The `25/zoom` visual effect is recovered
 *       by SHRINKING the hull path at the draw time via a second ctx scale.
 *       Too invasive for this phase.
 *
 * RECOMMENDED: variant (c) — zoom-bucket the cache. In practice zoom rarely
 * moves by more than a few buckets in a scroll burst, and buckets collide
 * during pan (zoom unchanged). The simplest implementation: `zoomBucket =
 * Math.round(zoom * 10) / 10`.
 */
export function getHullCache(
  nodes: GraphNode[],
  zoom: number,
  settledAt: number | null,
): Map<string, HullCacheEntry> {
  const zoomBucket = Math.round(zoom * 10) / 10;
  // Compose the epoch into a string key so null + any zoom bucket are distinct.
  const epoch = `${settledAt ?? 'null'}|${zoomBucket}` as unknown as number;
  if (epoch === cacheEpoch) return cache;

  cacheEpoch = epoch;
  cache = new Map();
  const byDir = new Map<string, GraphNode[]>();
  for (const n of nodes) {
    if (n.x === undefined || n.y === undefined) continue;
    if (n.dirKey === '') continue;
    const arr = byDir.get(n.dirKey) ?? [];
    arr.push(n);
    byDir.set(n.dirKey, arr);
  }

  for (const [dirKey, members] of byDir) {
    const pts = members.map((n) => [n.x!, n.y!] as [number, number]);
    const padded = paddedHullPoints(pts, 25 / zoom);
    const hull = polygonHull(padded);
    if (hull && hull.length >= 3) {
      const pathStr = smoothHullLine(hull);
      const smoothPath = pathStr ? new Path2D(pathStr) : null;
      const [cx, cy] = polygonCentroid(hull);
      cache.set(dirKey, {
        smoothPath,
        cx,
        cy,
        isCircleFallback: false,
        dirDepth: members[0].dirDepth,
      });
    } else {
      const cx = members.reduce((s, n) => s + (n.x ?? 0), 0) / members.length;
      const cy = members.reduce((s, n) => s + (n.y ?? 0), 0) / members.length;
      cache.set(dirKey, {
        smoothPath: null,
        cx,
        cy,
        isCircleFallback: true,
        dirDepth: members[0].dirDepth,
      });
    }
  }
  return cache;
}

/** Test-only — force a cache eviction. */
export function _resetHullCacheForTest(): void {
  cacheEpoch = -1 as number | null;
  cache = new Map();
}
```

## Pattern 4: Rolling-p95 Diagnostic

Thirty lines, inline in `RadarCanvas.tsx`'s main rAF render loop:

```ts
// Inside the RadarCanvas component body, near the other refs:
const perfRingRef = useRef<Float32Array>(new Float32Array(120)); // 120 frames = ~2s @ 60fps
const perfIdxRef = useRef(0);
const perfFilledRef = useRef(0);
const perfLogTimerRef = useRef(0);
const perfDebugEnabledRef = useRef(false);

// Read the flag once at mount (cheap; re-read not needed).
useEffect(() => {
  try {
    perfDebugEnabledRef.current = localStorage.getItem('radarPerfDebug') === '1';
  } catch {
    perfDebugEnabledRef.current = false; // private-browsing fallback
  }
}, []);

// ...inside the render() function in the main rAF loop, at the VERY TOP:
function render() {
  if (!ctx) return;
  if (!dirtyRef.current) {
    animFrameRef.current = requestAnimationFrame(render);
    return;
  }
  const t0 = perfDebugEnabledRef.current ? performance.now() : 0;

  // ...existing render body unchanged...

  if (perfDebugEnabledRef.current) {
    const dt = performance.now() - t0;
    const ring = perfRingRef.current;
    ring[perfIdxRef.current] = dt;
    perfIdxRef.current = (perfIdxRef.current + 1) % ring.length;
    if (perfFilledRef.current < ring.length) perfFilledRef.current++;
    // Emit every 60 frames (~1s). Avoid console spam.
    if (perfIdxRef.current === 0) {
      const n = perfFilledRef.current;
      const sorted = Array.from(ring.subarray(0, n)).sort((a, b) => a - b);
      const p95 = sorted[Math.min(n - 1, Math.floor(n * 0.95))];
      const max = sorted[n - 1];
      const avg = sorted.reduce((s, v) => s + v, 0) / n;
      console.log(
        `[radarPerf] p95=${p95.toFixed(2)}ms max=${max.toFixed(2)}ms avg=${avg.toFixed(2)}ms n=${n}`,
      );
    }
  }

  dirtyRef.current = false;
  animFrameRef.current = requestAnimationFrame(render);
}
```

**Bracket placement:** from just before the ctx operations (after the `if (!dirtyRef.current) return` fast-path so we only time real redraws) to just before `dirtyRef.current = false`. The trailing `requestAnimationFrame(render)` is NOT part of the measurement — it's the scheduling primitive itself.

**Why Float32Array (not a regular array)?** Avoids per-push allocations during long running sessions, keeps the p95 computation GC-free except for the once-per-60-frame sort array. Total overhead when the flag is OFF: one boolean read per frame.

**`localStorage` flag name:** `radarPerfDebug` (per D-12 spec). Enable via DevTools console: `localStorage.radarPerfDebug = '1'; location.reload()`. Reload is required — the flag is read once at mount.

**Optional HUD (Claude's discretion):** replace `console.log` with a ref-updating HUD text node. Adds ~30 more lines and one `<div>` with `position: fixed`. Recommend deferring to Phase 11.2 unless the planner feels strongly.

## Pitfalls

1. **Tab-sleep accumulator drift.** rAF pauses when the tab is hidden `[CITED: MDN]`. `pendingWheelRef` may accumulate state across the sleep if the tab goes hidden mid-burst. Acceptable per research §Pattern 1. If it ever proves problematic, add `document.addEventListener('visibilitychange', () => { pendingWheelRef.current = null; })` in the hook.
2. **React re-render timing for the D-06 writeback.** After D-01 lands, `setViewport` fires at most 1×/rAF. React 19 synchronously batches, so the render + `useEffect([viewport])` fires at most once per rAF. A defensive rAF coalescer on the writeback is harmless but not load-bearing. **Do not make claims about "cap minimap re-renders at 60 Hz" in the commit message for Task 2 if D-01 is already landed** — it's already capped by D-01.
3. **Test determinism — `renderHook` + fake timers.** `vi.useFakeTimers({ toFake: ['requestAnimationFrame'] })` must be set BEFORE `renderHook` is called in the same `it` block. Move it to `beforeEach`. `[CITED: vitest.dev/guide/mocking/timers]`.
4. **`cancelAnimationFrame` on unmount — one missed flush.** The `flushWheel` callback grabs `pendingWheelRef` inside the rAF body. If the unmount cleanup cancels the rAF AND nulls the pending ref, the pending events are dropped on unmount. That is the correct behavior — the component is gone, the events are moot.
5. **Zoom-bucket cache churn during fast zoom bursts.** At 0.1 bucket granularity, a full zoom sweep (0.05 → 20) crosses ~200 buckets. If the user zooms all the way in in ~0.5s, that's 400 cache rebuilds per second. STILL cheaper than rebuilding every frame on a 5k-node 200-directory graph (~200 polygonHull calls × 60fps = 12k calls/s). The cache wins even under pathological zoom. If a future profiling pass shows bucket churn is dominant, increase bucket granularity (0.2 or 0.5) or move to world-space-constant padding.
6. **Path2D polyfill in tests.** The existing test files inject a Path2D stub `[VERIFIED: GraphRenderer.test.ts:8-10, RadarCanvas.test.tsx:8-10]`. The new `hullCache.test.ts` should repeat the same shim — the cache calls `new Path2D(pathStr)` inside its rebuild.
7. **`setViewport` state identity in the spy assertion.** React's `setState` setter is a stable reference across renders, but `vi.spyOn(result.current, 'setViewport')` replaces it on the current snapshot only. Prefer behavior assertions (`viewport.zoom` changed) over call-count assertions for setter-spy fragility.
