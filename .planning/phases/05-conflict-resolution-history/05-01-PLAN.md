```sql
-- Conflict resolution records (HIST-02, D-11)
CREATE TABLE IF NOT EXISTS conflict_resolutions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    conflict_event_id INTEGER REFERENCES conflict_events(id),
    file_path TEXT NOT NULL,
    agent_a_id TEXT NOT NULL,
    agent_b_id TEXT NOT NULL,
    resolution_type TEXT NOT NULL CHECK(resolution_type IN ('accept_a', 'accept_b', 'manual', 'mixed')),
    backup_base_path TEXT,
    backup_a_path TEXT,
    backup_b_path TEXT,
    backup_merged_path TEXT,
    hunk_resolutions TEXT, -- JSON: [{hunkIndex, choice, customContent?}]
    notification_status TEXT DEFAULT 'pending',
    resolved_at TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Session file tracking junction table (D-09, HIST-01)
CREATE TABLE IF NOT EXISTS session_files (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id INTEGER NOT NULL REFERENCES agent_sessions(id),
    file_path TEXT NOT NULL,
    write_count INTEGER NOT NULL DEFAULT 1,
    last_written_at TEXT NOT NULL DEFAULT (datetime('now')),
    UNIQUE(session_id, file_path)
);

CREATE INDEX idx_session_files_session ON session_files(session_id);
CREATE INDEX idx_session_files_path ON session_files(file_path);
CREATE INDEX idx_conflict_resolutions_event ON conflict_resolutions(conflict_event_id);

-- Add file_count to agent_sessions for quick access (D-09)
ALTER TABLE agent_sessions ADD COLUMN file_count INTEGER NOT NULL DEFAULT 0;

-- Add resolution_id to conflict_events for linking
ALTER TABLE conflict_events ADD COLUMN resolution_id INTEGER REFERENCES conflict_resolutions(id);
```

Define serializable types (all with `#[derive(Debug, Clone, Serialize, Deserialize, Type)]` and `#[serde(rename_all = "camelCase")]`):

```rust
pub struct ConflictFileVersions {
    pub base_content: String,
    pub agent_a_content: String,
    pub agent_b_content: String,
    pub file_path: String,
    pub agent_a_id: String,
    pub agent_b_id: String,
}

pub struct HunkResolution {
    pub hunk_index: u32,
    pub choice: String, // "a", "b", "custom"
    pub custom_content: Option<String>,
}

pub struct ResolutionRecord {
    pub id: i64,
    pub conflict_event_id: Option<i64>,
    pub file_path: String,
    pub agent_a_id: String,
    pub agent_b_id: String,
    pub resolution_type: String,
    pub hunk_resolutions: String, // JSON
    pub notification_status: String,
    pub resolved_at: String,
}
```

Implement Tauri commands (all `#[tauri::command] #[specta::specta]`):

- `read_conflict_files(conflict_id: String, state: State<ConflictState>) -> Result<ConflictFileVersions, String>`: Looks up ConflictAlert by ID from ConflictState. Reads current file from disk. Gets base via `git show HEAD:<relative_path>` using `tokio::process::Command`. For agent_a and agent_b content, reads the file from disk (both agents wrote to the same file -- current state represents the latest write). File size capped at 1MB (1_048_576 bytes) -- return error "FILE_EXCEEDS_LIMIT" if larger.

- `apply_resolution(conflict_id: String, merged_content: String, hunk_resolutions: Vec<HunkResolution>, resolution_type: String, state: State<ConflictState>, pool: State<SqlitePool>, app: AppHandle) -> Result<ResolutionRecord, String>`:
  (1) Look up conflict alert.
  (2) Read current file + git base.
  (3) Use BackupManager to save base, agent_a, agent_b versions.
  (4) Write merged_content to the file path on disk.
  (5) Save backup of merged version.
  (6) INSERT into conflict_resolutions.
  (7) UPDATE conflict_events SET resolution_id.
  (8) Dismiss the conflict alert.
  (9) **Notify agents per D-12:** For each agent (agent_a_id, agent_b_id), send a chat message via the Phase 4 comms infrastructure. Use the same DB pool to INSERT into chat_messages directly (same pattern as `send_chat_message` in `comms::commands`): `INSERT INTO chat_messages (agent_id, direction, content, delivery_status) VALUES (?, 'outbound', ?, 'queued')`. Content: `"Conflict resolved on {file_path}: resolution type '{resolution_type}'. Your changes were {accepted/merged}."` Then UPDATE `conflict_resolutions SET notification_status = 'delivered'` for the new record. Emit "chat-message-sent" Tauri event for each message so the Comms UI updates.
  (10) Emit "conflict-resolved" Tauri event.
  (11) Return the ResolutionRecord (with notification_status reflecting delivery).

- `list_conflict_resolutions(pool: State<SqlitePool>) -> Result<Vec<ResolutionRecord>, String>`: SELECT from conflict_resolutions ORDER BY resolved_at DESC.

- `list_session_files(session_id: i64, pool: State<SqlitePool>) -> Result<Vec<SessionFileRecord>, String>`: SELECT from session_files WHERE session_id ORDER BY write_count DESC LIMIT 10.

- `record_session_file(session_id: i64, file_path: String, pool: State<SqlitePool>) -> Result<(), String>`: INSERT INTO session_files ... ON CONFLICT(session_id, file_path) DO UPDATE SET write_count = write_count + 1. Also UPDATE agent_sessions SET file_count = (SELECT COUNT(*) FROM session_files WHERE session_id = ?).

- `list_sessions(pool: State<SqlitePool>) -> Result<Vec<SessionRecord>, String>`: SELECT from agent_sessions with file_count, ORDER BY started_at DESC.

- `list_approval_history(pool: State<SqlitePool>) -> Result<Vec<ApprovalHistoryRecord>, String>`: SELECT from approval_requests with joins, ORDER BY resolved_at DESC. **Note (HIST-03):** This reads from the Phase 4 `approval_requests` table which already stores all approval decision data. No additional storage migration is needed for HIST-03.

Add `SessionRecord`, `SessionFileRecord`, and `ApprovalHistoryRecord` types with appropriate fields matching the existing schema + new columns.

Validate file_path in `read_conflict_files` and `apply_resolution`: reject paths containing `..` segments.

**2. No Cargo.toml changes needed** -- `tokio::process::Command` is already available via tokio with full features, and `sqlx` is already a dependency.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Filesystem read/write | Conflict resolution reads files from disk and writes merged content back |
| Git subprocess | Spawns `git show HEAD:<path>` to read base file version |
| IPC (frontend -> Rust) | Frontend sends conflict_id, merged_content, file_path via invoke |
| Comms delivery (D-12) | Resolution notification sent as chat message to involved agents |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Tampering | resolution.rs read_conflict_files | mitigate | Validate file_path contains no ".." segments before reading. Cap file size at 1MB. |
| T-05-02 | Tampering | backup.rs save_backup | mitigate | Validate conflict_id and label contain no path separators or ".." segments. Resolve paths within backup_dir only. |
| T-05-03 | Tampering | resolution.rs apply_resolution | mitigate | Validate file_path from ConflictAlert (server-side data, not user input). Reject paths with ".." segments. Write only to paths within the watched directory. |
| T-05-04 | Denial of Service | resolution.rs read_conflict_files | mitigate | Cap file size at 1MB (1_048_576 bytes). Return FILE_EXCEEDS_LIMIT error for larger files. |
| T-05-05 | Information Disclosure | backup.rs read_backup | mitigate | Validate relative_path resolves within backup_dir. No directory traversal. |
| T-05-06 | Spoofing | Git subprocess | accept | Git command runs locally with user's credentials. Low risk for single-user desktop app. |
</threat_model>
