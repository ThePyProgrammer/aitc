`BRIDGE_HIT_RADIUS` **auto-selected as 10px** (vs file node's 8px): diamond geometry at 8px half-diagonal has an 8×8 bounding box but a smaller hit surface at the corners; 10px hit radius keeps fingertip slop consistent with file hover. Quadtree hit-test order: file nodes first, bridges second (bridges always pass through the file-position pass untouched because they're on `y=0` which is free of file nodes by construction — the forceBoundary pushes files away).

### Canvas-rendered bridge label

World-space bridge name (visible at `zoom ≥ 4`, matching `FILE_LABEL_ZOOM_THRESHOLD`):

```js
ctx.font = `${10 / zoom}px "JetBrains Mono", monospace`;
ctx.fillStyle = theme.fileLabelColor;         // reuse the same token as file labels
ctx.globalAlpha = 0.9;                         // slightly higher than file labels so bridges pop
ctx.fillText(commandName, bridge.x, bridge.y - (BRIDGE_HALF_DIAG + BRIDGE_LABEL_OFFSET) / zoom);
ctx.globalAlpha = 1;
```

Below `zoom < 4`, bridges render unlabeled per CONTEXT D-18 — the tooltip carries label info at any zoom.

### FRONTEND / BACKEND label positioning math

```ts
// Draw after the world-space pass, inside ctx.save() + ctx.setTransform(1,0,0,1,0,0)
const boundaryScreenY = viewport.panY;  // world y=0 projected: y_screen = 0*zoom + panY = panY
const leftX = 12;                        // 12px screen inset
// FRONTEND main label anchors at (leftX, boundaryScreenY - 8 - 10) = 18px above boundary
// (TypeScript sub-line) anchors at (leftX, boundaryScreenY - 8) = 8px above boundary
// BACKEND main label anchors at (leftX, boundaryScreenY + 8 + 10)
// (Rust sub-line) anchors at (leftX, boundaryScreenY + 8)
```

When the boundary is off-screen (user panned far up or down), labels **clamp** to viewport edges (auto-selected): if `boundaryScreenY < 0`, stick both label stacks to `y = 24` (top edge); if `boundaryScreenY > canvas.height`, stick both to `y = canvas.height - 24`. Clamping keeps the semantic signal alive even when the user has zoomed into one hemisphere — they see which side of the boundary they're on without needing to pan back to the line.

## Interaction Contracts

### Bridge hover

- Cursor: `pointer` over a bridge (hit-test against all bridges after file nodes in the quadtree pass — priority order auto-selected as "files first, bridges second," rationale: bridges are deterministic-x and lie on y=0, so file hits are almost always mutually exclusive; files-first keeps the existing Phase 7 quadtree intact without forcing the bridge index to dominate).
- Bridge diamond: **no growth animation** (auto-selected — see Motion).
- After **200ms dwell**, `BridgeTooltip` appears above+right of cursor with:
  - Line 1 (Data 12/700): `{commandName}` (camelCase, as it appears in bindings.ts).
  - Line 2 (Label 10/400 UPPERCASE, subtle): `{rust_name}` (snake_case Rust name).
  - Line 3 (Data-sm, mono, label+value): `HANDLER {handlerFile}:{handlerLine}` — truncated with ellipsis if > 40 chars.
  - Line 4 (Data-sm, mono, label+value): `{callerCount}_CALLERS` (e.g., `3_CALLERS`, `0_CALLERS` for dangling).
  - Line 5 (Data-sm, mono, label+value, wraps): `{signatureSummary}`.
  - Line 6 (Label 10/400 UPPERCASE, cyan — conditional): `CHANNEL-BEARING` — rendered only when `hasChannelArg`.
  - Line 7 (Label 10/400 UPPERCASE, `on-surface-variant` at 50% — conditional): `DANGLING — NO CALLERS` or `DANGLING — NO HANDLER` — rendered only for dangling bridges.
- Tooltip position: offset +12px right, +12px down from cursor; clamps to viewport edges (inherits `AgentTooltip` clamp logic).

### Bridge click (new selection slot)

- Single-click bridge: `radarStore.selectBridge(commandName)` → sets `selectedBridgeId`. No-op if clicking the already-selected bridge (use the close [X] button to deselect — consistent with manifest idiom).
- Visual response: bridge gains 1px white outer ring at 80% alpha (screen-space unchanged, `1/zoom` width). Right-side manifest panel scrolls to the bridge detail section (if not already visible) and fades the section in (150ms).
- Selecting a bridge does NOT deselect the currently-selected agent. Both can coexist. (Auto-selected per parallelism with Phase 7 where `selectedAgentId` and `hoveredNodeId` coexist; F-01 defers focus-mode which would make selection exclusive.)

### Bridge tooltip dismissal

- On `mouseleave` of bridge: tooltip fades out over 150ms. If the user moves onto another bridge during the fade, tooltip snaps to the new bridge (no re-dwell for same-frame transitions between adjacent bridges).

### Bridge detail panel — caller row click

- Click a caller path row: pans main viewport to caller file's world position at zoom 3× (matches `AgentManifestRow` idiom). 1.5s `drawLeadLine` from bridge to caller. Does NOT change `selectedBridgeId`.
- Row hover: `surface-container-high` background fill (matches agent row hover).

### Bridge detail panel — close button

- Label: `CLOSE` (Label 10/400 UPPERCASE, placed at the section heading's right edge like an X affordance — auto-selected as textual `CLOSE` over an icon-only X button, rationale: ATC UPPERCASE aesthetic is consistent).
- Icon auto-selected: Lucide `X`, 14px, strokeWidth 1.5px, left of the `CLOSE` text. Pattern matches existing component iconography.
- Click: `selectBridge(null)`. Panel unmounts with 150ms opacity fade.

### FRONTEND / BACKEND label — interactive?

- **Non-interactive.** Auto-selected: hovering the label does nothing. Clicking does nothing. The labels are chrome, not buttons. If a future phase wants a "filter to frontend only" toggle, it should be a new affordance next to the HEAT_MAP button.

### Pan / zoom

- **Unchanged from Phase 7.** Boundary line pans and scales with the world. Bridges are world-space and travel with pan/zoom. FRONTEND/BACKEND labels remain anchored to the screen's left edge; their y follows the projected boundary-y (with clamping to viewport edges when boundary is off-screen, per Layout §).

### Keyboard shortcuts (new)

Inherited Phase 7 hotkeys (`F`, `H`, `M`, `Escape`) carry over. One new auto-selected addition:

- **`B`**: toggle visibility of bridges + boundary line (all bridges + line hide; file graph renders as-is). Auto-selected rationale: gives user a clean way to A/B between "with IPC layer" and "without IPC layer" without touching force sliders. Stored as `radarStore.bridgesVisible: boolean` (default `true`). This extends keyboard hotkey model symmetrically with `H` for heat-map.

`Escape` deselects bridge in addition to its existing deselect-agent behavior. Both can be deselected in one press (auto-selected — matches user's mental model of "escape = clear all selection").

### Pipeline-driven updates

- Per CONTEXT D-23/D-24, `installRadarPipelineBridge`'s 500ms debounce already fires `fetchGraph()`. Bridges refresh along with the graph on any eligible file change. Planner optionally adds a path-filter predicate (`changed_paths ⊇ src/bindings.ts ∪ src-tauri/**/*.rs ∪ src/**/*.ts(x)`) to skip bridge re-parse when the change is irrelevant — **not** required for v1 (<100ms parse cost per CONTEXT D-35).

### Bridge node participation in existing mechanics

- **Heat map**: bridges tint with `heatColor(score)` if their path-key has a contention score. Auto-selected rationale: CONTEXT mandate "bridges are path-keyed like file nodes and participate in the same spatial hit-testing." Heat shows when a command is heavily invoked by agents (interesting emergent signal).
- **Comet trails**: file-to-bridge trails spawn when an agent's most-recent file event is on a caller file that then writes/reads the bridge's handler file — auto-selected to **NOT emit** (per CONTEXT §Deferred: agent-driven invoke animation is v2). v1 agent comet trails continue to travel between file nodes only.
- **Conflict pulse**: a bridge whose handler file is in conflict pulses red like any file node with a conflict. Auto-selected — same contract.
- **Pinning**: bridges CANNOT be pinned by user drag in v1 (CONTEXT §Deferred "drag-to-pin bridges — requires abandoning alphabetic deterministic layout"). Hit-test on mousedown over a bridge → no drag initiation. File-node drag-to-pin unchanged.
